# Quick fix — alinhar altura de Número e Complemento

## Diagnóstico (já investigado, reproduzido localmente)
O label de "Complemento" (`#billing_address_2_field label.screen-reader-text`)
foi desescondido num fix anterior com `font-size: 12px !important`, menor que
o label padrão de "Número" (sem override, herda o tamanho default do tema,
maior que 12px). Isso faz os dois campos terem alturas de conteúdo (label +
input) diferentes — e como não há alinhamento vertical explícito entre os
dois campos na mesma linha, os inputs ficam desalinhados (confirmado com
screenshot local em viewport mobile).

## Por que não vou fazer o wrap em uma nova div
A sugestão de colocar os dois num `<div>` funcionaria, mas exigiria
manipulação de DOM via JS (mover dois elementos `<p>` irmãos do WooCommerce
para dentro de um wrapper novo) — mais uma peça móvel, e o projeto já teve um
caso (página 001) onde reestruturar DOM via JS/CSS não-trivial quebrou outra
coisa (o widget do Mercado Pago). Existe uma solução puramente CSS, mais
simples e no mesmo padrão já usado nesse arquivo pra esse tipo de problema:
`align-self: flex-end` nos dois campos, que já são itens flex dentro do
wrapper `.woocommerce-billing-fields__field-wrapper` (isso não desce dos
outros pares — Tipo de Pessoa+CPF, Cidade+Estado — que continuam com o
comportamento atual).

## Mudança proposta
Em `templates/checkout-elementor-data.json`, no `custom_css` do widget
`bd9deff`, adicionar (perto das outras regras de pares de campos):
```css
selector #billing_number_field,
selector #billing_address_2_field {
	align-self: flex-end;
}
```
Isso alinha os dois inputs pela base (bottom), independente da diferença de
altura dos labels acima deles — sem mexer em DOM, sem JS novo.

## Escopo
- Só `templates/checkout-elementor-data.json` (custom_css do widget bd9deff).
- Não mexe em `#billing_persontype_field`/`#billing_cpf_field` nem
  `#billing_city_field`/`#billing_state_field` (outros pares, não afetados).
